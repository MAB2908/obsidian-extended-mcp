# Четырёхуровневая система разработки проектов
## Инструкция (System Prompt) для AI-ассистированной разработки

> **Автор концепции:** Boris Cherny (Anthropic, создатель Claude Code)  
> **Адаптация:** практическая инструкция для применения в проектах любого масштаба

---

## Мета-принцип

Не спрашивай: *«Как получить от ИИ лучший ответ?»*  
Спрашивай: *«Как построить систему, в которой ИИ надёжно даёт то, что нужно?»*

Разница между «использовать инструмент» и «инженерить с помощью инструмента».

---

## Уровень 1. PROMPT — Единичный запрос

### Правила формулирования

1. **Outcome-based prompting** — формулируй задачу через результат, а не действия:
   - ❌ «Напиши тесты для модуля X»
   - ✅ «Исправь упавшие CI-тесты и добейся зелёного пайплайна»

2. **Challenge-режим** — требуй рефлексии, а не только генерации:
   - «Разбери мои изменения и докажи, что они работают корректно»
   - «Найди 3 потенциальных бага в этом решении до того, как я их вмержу»

3. **Голосовой диктейт** — если возможно, диктуй промпт голосом (в 3 раза быстрее печати), затем редактируй. Детальные промпты дают детальные результаты.

4. **Контекст-гигиена** — не используй контекст как мусорку. Перед каждым запросом:
   - Удали устаревшие файлы из контекста
   - Добавь только релевантные `CLAUDE.md` или `SKILL.md`
   - Если контекст >50% токенов — перезапусти сессию

### Шаблон промпта

```
[РОЛЬ]
Ты — {специализация}. Твоя задача — {результат}.

[КОНТЕКСТ]
- Проект: {название}
- Стек: {технологии}
- Статус: {что уже сделано / что сломано}
- Ограничения: {что нельзя менять}

[ЗАДАЧА]
{конкретный результат, который нужен}

[КРИТЕРИИ ПРИЁМКИ]
1. {измеримый критерий 1}
2. {измеримый критерий 2}
3. {измеримый критерий 3}

[ПРОВЕРКА]
Перед ответом:
- Запусти {команда проверки}
- Убедись, что {условие успеха}
- Если проверка не пройдена — исправь и повтори
```

## Уровень 2. SKILL — Повторяющийся навык

### Когда создавать Skill

Если действие повторяется **более 1 раза в день** — преврати его в Skill.

### Структура Skill-файла

Файл: `.claude/skills/{имя-скилла}.md` (проектный) или `~/.claude/skills/{имя-скилла}.md` (персональный)

```markdown
---
name: {имя-скилла}
description: {что делает}
permissions:
  - {команда 1}: {pre-approved / ask / deny}
  - {команда 2}: {pre-approved / ask / deny}
---

# {Имя скилла}

## Контекст
{Когда и зачем использовать}

## Предусловия
- [ ] {что должно быть true до запуска}
- [ ] {что должно быть true до запуска}

## Пошаговый алгоритм
1. {Шаг 1}
2. {Шаг 2}
3. {Шаг 3}

## Постусловия / Проверка
- [ ] {команда проверки 1}
- [ ] {команда проверки 2}

## Примеры
### Успешный сценарий
{пример входных данных и ожидаемый результат}

### Ошибка и обработка
{типичная ошибка + как исправить}
```

### Примеры скиллов для проекта

| Skill | Назначение |
|-------|-----------|
| `/commit-push-pr` | Коммит, пуш, создание PR с описанием |
| `/techdebt` | Поиск дублирующегося кода и устаревших зависимостей в конце сессии |
| `/simplify` | Параллельный ревью кода: 3 агента ищут избыточность |
| `/verify` | Запуск тестов + линтер + type-check + скриншот-тесты |
| `/onboard` | Объяснение новому разработчику структуры проекта за 5 минут |

## Уровень 3. AGENT — Специализированный агент

### Принцип: один агент — одна роль

Не создавай универсального агента. Создавай **флот специалистов**.

### Типы агентов в проекте

```yaml
agents:
  code-reviewer:
    role: "Читает код и даёт фидбек"
    tools: [read, grep, diff]
    constraints: ["не пишет код", "не запускает команды"]

  test-writer:
    role: "Пишет и исправляет тесты"
    tools: [read, write, bash]
    constraints: ["только test/ директория", "не трогает src/ без разрешения"]

  verify-app:
    role: "Проверяет работу через внешние интерфейсы"
    tools: [browser, curl, bash]
    constraints: ["только проверка", "не правит код"]

  code-simplifier:
    role: "Упрощает код после реализации"
    tools: [read, write, diff]
    constraints: ["сохраняет поведение", "не добавляет функциональность"]

  architect:
    role: "Проектирует структуру перед реализацией"
    tools: [read, write, grep]
    constraints: ["только .md и .yaml файлы", "не пишет бизнес-логику"]
```

### Параллельная работа агентов

Запускай 5–15 агентов одновременно, каждый в своём git worktree:

```bash
# Создание изолированных worktrees для параллельных агентов
git worktree add ../agent-review-main review-branch
git worktree add ../agent-test-main test-branch
git worktree add ../agent-simplify-main simplify-branch
```

Каждый агент работает в своём контексте и не мешает другим.

## Уровень 4. WORKFLOW — Оркестрация

### Фазовый пайплайн (Spec → Draft → Simplify → Verify)

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│   SPEC      │ -> │   DRAFT     │ -> │  SIMPLIFY   │ -> │   VERIFY    │
│  (Plan)     │    │  (Implement)│    │  (Refactor) │    │  (Validate) │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
     │                   │                   │                   │
     ▼                   ▼                   ▼                   ▼
  architect          code-writer        code-simplifier      verify-app
  + product-owner    + test-writer      + code-reviewer      + test-writer
```

### Правила перехода между фазами

| Фаза | Выходной артефакт | Критерий перехода |
|------|-------------------|-------------------|
| **Spec** | `spec.md` + `plan.md` | Человек аппрувит план |
| **Draft** | Рабочий код + тесты | CI зелёный, линтер чист |
| **Simplify** | Упрощённый код | Code-review пройден, дубли удалены |
| **Verify** | Подписанный релиз | Все проверки пройдены, `CLAUDE.md` обновлён |

### Живая документация: CLAUDE.md

Файл `.claude/CLAUDE.md` — это память проекта. После каждой ошибки добавляй правило.

```markdown
# Правила проекта {название}

## Архитектура
- {ключевое архитектурное решение 1}
- {ключевое архитектурное решение 2}

## Код-стайл
- {специфичное правило, не покрытое линтером}
- {специфичное правило, не покрытое линтером}

## Частые ошибки (DO NOT REPEAT)
- ❌ {что делал неправильно} → ✅ {как правильно}
- ❌ {что делал неправильно} → ✅ {как правильно}

## Рабочие процессы
- {как запускать проект локально}
- {как деплоить}
- {как откатывать при ошибке}

## Контекстные файлы
- `src/core/` — критичный код, трогать только через `/architect`
- `test/e2e/` — долгие тесты, запускать только в `/verify`
```

### Автоматизация через Hooks

```yaml
# .claude/config.yaml
hooks:
  PostToolUse:
    - tool: bash
      command: "npm run format && npm run lint --fix"
      condition: "file_extension in ['.js', '.ts', '.tsx']"

  PreApprove:
    - pattern: "git commit -m"
      action: ask
    - pattern: "npm test"
      action: auto
    - pattern: "rm -rf"
      action: deny
```

## Чек-лист запуска нового проекта

### Перед первой задачей
- [ ] Создан `.claude/CLAUDE.md` с базовыми правилами
- [ ] Создана директория `.claude/skills/` с минимум 3 скиллами
- [ ] Настроены permissions в `.claude/config.yaml`
- [ ] Определены роли агентов и их constraints
- [ ] Настроены git worktree для параллельной работы

### Перед каждой сессией
- [ ] Контекст очищен от мусора (<50% токенов)
- [ ] Актуальные `CLAUDE.md` и `SKILL.md` загружены
- [ ] Plan Mode включён для задач >30 минут реализации
- [ ] Verification Loop определён (как проверить результат)

### После каждой сессии
- [ ] `CLAUDE.md` обновлён новыми ошибками/правилами
- [ ] Новые повторяющиеся действия вынесены в Skills
- [ ] Результат верифицирован (тесты, линтер, ручная проверка)
- [ ] Git worktree очищен

## Анти-паттерны (чего НЕ делать)

| ❌ Анти-паттерн | ✅ Правильный подход |
|----------------|----------------------|
| Один универсальный агент на всё | Флот специализированных агентов |
| Писать длинные промпты каждый раз | Вынести в Skill и вызывать по имени |
| Использовать контекст как мусорку | Регулярно чистить, перезапускать сессии |
| Просить «напиши код» без проверки | Требовать verification loop в каждой задаче |
| Править баги вручную | Добавлять правило в `CLAUDE.md`, чтобы агент не повторял |
| Работать последовательно | Параллелить независимые задачи через worktree |

## Пример: полный цикл задачи

**Задача:** Добавить систему аутентификации

```
1. SPEC (architect + product-owner)
   → Пишем `spec/auth-spec.md`: OAuth2 vs JWT, схема БД, API endpoints
   → Пишем `plan/auth-plan.md`: 5 шагов реализации
   → Человек аппрувит

2. DRAFT (code-writer + test-writer)
   → Агент реализует по плану
   → Параллельно test-writer пишет тесты
   → verify-app проверяет API через curl
   → CI должен быть зелёным

3. SIMPLIFY (code-simplifier + code-reviewer)
   → 3 параллельных агента ревьюят код
   → Удаляем дубли, упрощаем абстракции
   → Проверяем, что тесты всё ещё проходят

4. VERIFY (verify-app + человек)
   → Полный regression test
   → Security check (SQL-injection, XSS)
   → Обновляем `CLAUDE.md`: «Auth использует JWT, не OAuth2»
   → Merge
```

## Заключение

Эта система масштабируется от одного разработчика до команды из 50 человек. Ключевые метрики успеха:

- **Время на задачу** сокращается за счёт Skills и параллелизма
- **Количество багов** падает за счёт Verification Loop и специализированных агентов
- **Onboarding** нового разработчика занимает часы, а не недели (благодаря `CLAUDE.md`)
- **Повторяемость** — одинаковые задачи решаются одинаково хорошо независимо от того, кто их выполняет

> *«Хороший план позволяет Claude сделать задачу в один проход. Хорошая система позволяет делать это надёжно, снова и снова.»* — Boris Cherny
